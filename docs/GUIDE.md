# meshping Guide

This guide matches the current simplified product surface.

For the formal requirements document, see [docs/SRS.md](SRS.md).

`meshping` is a terminal-first network monitor. It answers one question quickly:

"Is the network healthy, or is something else making my app feel slow?"

It does that with a live matrix, plain-English labels, a health score, local CPU/RAM correlation, and short diagnostic commands.

## Current Product Surface

Implemented and user-facing:

- `meshping` with no arguments: launches a safe sandbox session against Cloudflare, Google, and your local router.
- `meshping "check my dns and the router"`: natural-language quick start for common checks.
- `meshping probe`: one-shot latency checks.
- `meshping mesh`: live TUI for ongoing monitoring.
- `meshping top`: live TUI with local CPU/RAM/NIC correlation shown by default.
- `meshping diff`: side-by-side comparison of two targets.
- `meshping check`: script-friendly pass/fail health checks.
- `meshping agent` plus `meshping mesh --distributed`: full node-to-node distributed mesh.
- `meshping discover`: find running agents on a subnet.
- `meshping doctor`: local network diagnosis with recommendations.
- `meshping replay`: replay a recorded `.mpr` session.

Visible TUI features:

- 0-100 network health score
- split-brain diagnosis that says `WiFi interference` or `ISP congestion`
- IPv4 vs IPv6 protocol-race comparison per target
- usage profiles: `work`, `gaming`, `video`
- plain-English "What's Wrong?" labels
- service auto-naming
- first-run tutorial screen
- `?` contextual help
- local machine panel with CPU, RAM, loopback, and NIC counters
- heatmap view backed by in-memory session history
- clipboard share hotkey
- mute/snooze hotkey
- ISP/city banner when available

Simplified or removed from the user-facing surface:

- no Prometheus or Grafana exporter commands
- no manual `--budget` workflow
- no weather icon terminology
- no flood/burst stress mode in the main CLI
- no dedicated `stress` or `canary` commands in the main CLI
- no required SQLite file for normal TUI usage

## Quick Start

Run the safest default experience:

```bash
meshping
```

That opens a live TUI against:

- `cloudflare` (`1.1.1.1:443`)
- `google` (`8.8.8.8:443`)
- `router` (your default gateway on the first open port from `53`, `80`, or `443`)

Run a natural-language check:

```bash
meshping "check my dns and the router"
```

Run the live matrix against specific services:

```bash
meshping mesh backend=127.0.0.1:8080 postgres=127.0.0.1:5432 ollama=127.0.0.1:11434
```

Tune the scoring profile:

```bash
meshping mesh --profile gaming backend=127.0.0.1:8080 postgres=127.0.0.1:5432
meshping mesh --profile video cloudflare=1.1.1.1:443 google=8.8.8.8:443
```

Run the local resource correlation view:

```bash
meshping top backend=127.0.0.1:8080 postgres=127.0.0.1:5432
```

## Local Demo Stack

Use this when you want an easy presentation with a real 8x8 adjacency matrix on one laptop.

Terminal 1:

```bash
meshping demo
```

`demo` starts 8 tiny HTTP microservice simulators on localhost and opens the full adjacency matrix in the same process.
The real listeners are the HTTP ports `18080-18087`.
The 8x8 adjacency matrix is generated by the built-in reporter so you can demo node-to-node latency on one laptop.

```text
frontend  backend  auth  orders  payments  inventory  redis  postgres
```

## Core Commands

### `meshping probe`

Use for a fast one-shot check.

```bash
meshping probe backend=127.0.0.1:8080 postgres=127.0.0.1:5432
meshping probe cloudflare=1.1.1.1:443 google=8.8.8.8:443
```

### `meshping mesh`

Use for ongoing monitoring in the TUI.

```bash
meshping mesh backend=127.0.0.1:8080 postgres=127.0.0.1:5432
meshping mesh --nodes backend=127.0.0.1:8080,postgres=127.0.0.1:5432 --record incident.mpr
meshping mesh --nodes backend=127.0.0.1:8080,postgres=127.0.0.1:5432 --log ~/.meshping/session.db
meshping mesh --once backend=127.0.0.1:8080 postgres=127.0.0.1:5432
```

Notes:

- normal session history stays in memory
- add `--log` only when you want a SQLite file
- add `--record` when you want replay later

### `meshping top`

Use when you want latency and local machine health together.

```bash
meshping top
meshping top backend=127.0.0.1:8080 postgres=127.0.0.1:5432
```

This helps answer:

"Is the network slow, or is my laptop overloaded?"

### `meshping diff`

Use when you want a clean comparison between two paths.

```bash
meshping diff cloudflare=1.1.1.1:443 google=8.8.8.8:443
meshping diff router=192.168.1.1:80 dns=8.8.8.8:443
```

### `meshping check`

Use in scripts or CI.

```bash
meshping check --nodes backend=127.0.0.1:8080,postgres=127.0.0.1:5432 --samples 10 --max-p99 50 --max-loss 1
```

Exit code:

- `0`: healthy enough
- `1`: threshold exceeded or no usable samples

### `meshping discover`

Use when you want to find running meshping agents on a local subnet.

```bash
meshping discover --subnet 192.168.1.0/24
meshping discover --subnet 10.0.0.0/24 --port 7777 --timeout 1.0
```

## Distributed Mode

Use distributed mode when you want a true source-to-target matrix between multiple nodes.

Start one agent per node:

```bash
meshping agent --name frontend --host 0.0.0.0 --port 7777
meshping agent --name postgres --host 0.0.0.0 --port 7778
meshping agent --name ollama --host 0.0.0.0 --port 7779
```

Run the coordinator:

```bash
meshping mesh --distributed --agents "frontend=10.0.0.11:7777,postgres=10.0.0.12:7778,ollama=10.0.0.13:7779" --listen 0.0.0.0:7780
```

You can also provide the same list through the environment:

```bash
export MESHPING_AGENTS="frontend=10.0.0.11:7777,postgres=10.0.0.12:7778,ollama=10.0.0.13:7779"
meshping mesh --distributed --listen 0.0.0.0:7780
```

## `meshping doctor`

Use for a short diagnosis with plain-English fixes.

```bash
meshping doctor
meshping doctor --ports 80,443,5432,8080
meshping doctor --output doctor-report.txt
```

Checks include:

- loopback
- default gateway
- split-brain blame check
- DNS
- internet reachability
- IPv6
- MTU behavior
- TCP retransmits
- NIC drops
- outbound ports

## `meshping replay`

Replay a recorded session:

```bash
meshping replay incident.mpr
meshping replay incident.mpr --speed 2x
meshping replay incident.mpr --seek 00:01:30
meshping replay incident.mpr --once
```

Replay keys:

- `SPACE`: pause or resume
- `LEFT`: slower
- `RIGHT`: faster
- `g`: jump to `HH:MM:SS`
- `q`: quit

## TUI Keys

Inside `meshping` / `meshping mesh` / `meshping top`:

- `q`: quit
- `s`: stats panel
- `a`: alerts panel
- `c`: clusters
- `h`: heatmap
- `l`: local machine panel
- `Enter`: time-series
- `J`: histogram
- `T`: cluster topology
- `S`: copy a human-readable support summary to clipboard
- `p`: cycle usage profile
- `M`: mute or unmute the selected target
- `E`: export selected heatmap as CSV
- `1`, `2`, `3`: refresh presets
- arrows: move selection
- `?`: contextual help

## Interpreting The Screen

The matrix uses three status levels:

- `GOOD` in green: healthy and stable
- `WARN` in yellow: latency or jitter is elevated
- `BAD` in red: loss, timeouts, or severe degradation

The "What's Wrong?" text explains the raw numbers in plain English, for example:

- `Good for gaming`
- `Good for calls`
- `Packet loss - check WiFi`
- `Buffer warning`
- `Slow path - check ISP/VPN`

The split-brain diagnosis works like this:

- gateway slow: blame local Wi-Fi or the router path
- gateway fast but `8.8.8.8` slow: blame the ISP or upstream path
- both healthy: local path is probably not the problem

The protocol race works like this:

- the tool quietly probes IPv4 and IPv6 when both exist
- if IPv6 is noticeably slower, the stats panel shows `IPv6 slower than IPv4`
- the chosen cell prefers the faster family, which helps expose Happy Eyeballs failures

The health score is a single 0-100 rollup. It drops when:

- latency rises
- jitter rises
- packet loss appears
- timeouts happen

## What Is Not In Scope

These are not part of the current simplified tool:

- Prometheus / Grafana exporter workflow
- manual budget definition in the CLI
- weather icon terminology
- AS-number analysis
- hop-by-hop route visualization
- flood/burst demo mode
- dedicated stress/canary load commands

pip install -e ".[dev]"
